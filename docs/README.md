# Telegram Shop Bot «Solid Simple»

**Версия:** 3.5 (последняя, март 2026)  
**Статус:** ✅ Production Ready  
**Платформа:** Telegram Bot API 9.4+  
**Язык:** Python 3.13 + python-telegram-bot 22.7

🎨 **Новое в ver. 3.5:**
- ✅ Все Reply кнопки цветные (100% устройств)
- ✅ Inline кнопки цветные на мобильных (80% пользователей)
- ⚪ Inline кнопки серые на Desktop (20% пользователей)
- 🔵🟢🔴 Полная поддержка стилей кнопок (primary/success/danger)

Телеграм-бот-магазин для продажи товаров с каталогом, корзиной, оформлением заказа, доставкой и оплатой.

## Для кого этот документ
- Владелец бизнеса / менеджер — без опыта программирования
- Новичок, который хочет запустить или доработать бот
- Разработчик, который будет поддерживать или развивать проект

## Что делает этот бот (простыми словами)
- Показывает каталог товаров в Telegram.
- Даёт добавить товары в корзину, изменить количество и удалить позиции.
- Ведёт человека через оформление заказа: контакты, доставка, комментарий, оплата.
- Поддерживает оплату через Telegram Payments и YooKassa.
- Может отправлять уведомления админам и клиенту.
- Использует сохранённые данные о товарах, картинках и текстах из файла `data/shop_data.json`.

Проще говоря: человек листает витрину в Telegram, выбирает товар, оформляет заказ и оплачивает его без отдельного сайта.

## Как это работает под капотом (для любопытных, но всё ещё понятно)
1. `main.py` запускает бота, инициализирует БД, корзину и регистрирует обработчики команд.
2. `handlers/` содержит логику кнопок, сообщений и шагов оформления заказа.
3. `core/database.py` хранит корзину, заказы и состояние пользователя в SQLite.
4. `core/checkout_flow.py` ведёт человека по шагам оформления заказа.
5. `core/telegram_payments.py` и `core/yookassa_webhook.py` отвечают за оплату и приём уведомлений об оплате.
6. На стороне сайта лежит PHP-вебхук в `solidsimple.ru/public_html/yookassa/webhook.php`.
7. Дополнительно есть AI-модули (`core/ai_assistant.py`, `core/gigachat_assistant.py`), но это не ядро магазина.

У проекта есть две связанные части: Telegram-бот и веб-контур для YooKassa. Это не один чистый monolith, а связка из нескольких слоёв.

## Установка и запуск (подробно, два варианта: локально и на сервере)

### Вариант 1. Локально

Это подходит для проверки, доработки и отладки.

```bash
python -m venv .venv
source .venv/bin/activate   # Linux / macOS
# .venv\Scripts\activate    # Windows
pip install python-telegram-bot python-dotenv httpx aiohttp requests PyMySQL yookassa cryptography
python main.py
```

Что важно проверить до запуска:
- `.env` заполнен без мусорных тестовых значений;
- `BOT_TOKEN` реальный, а не заглушка;
- `data/shop_data.json` существует;
- если нужны платежи, настроены YooKassa и Telegram Provider Token;
- если MySQL-часть используется, доступны `YOOKASSA_RELAY_DB_*`.

### Вариант 2. На сервере

Этот вариант нужен для постоянной работы.

1. Загрузите проект на сервер.
2. Установите Python 3.13+ и системные библиотеки для `venv`.
3. Создайте виртуальное окружение и поставьте зависимости, как в локальном варианте.
4. Скопируйте `.env` на сервер, но **не коммитьте** его в git.
5. Запускайте бота через `systemd`, `supervisor` или хотя бы `screen`/`tmux`.
6. Если используете PHP-вебхук, отдельно разверните каталог `solidsimple.ru/public_html` на веб-сервере.
7. Удалите опасные служебные файлы перед публикацией:
   - `info.php`
   - логи
   - `.bak`
   - отладочные дампы

Пример `systemd`-юнита:

```ini
[Unit]
Description=Solid Simple Telegram Bot
After=network.target

[Service]
WorkingDirectory=/opt/solid-simple
EnvironmentFile=/opt/solid-simple/.env
ExecStart=/opt/solid-simple/.venv/bin/python main.py
Restart=always
User=bot

[Install]
WantedBy=multi-user.target
```

## Важные предупреждения и известные проблемы

### 🔒 Критические изменения в ver. 3.2:

**Голосовой режим ПОЛНОСТЬЮ УДАЛЁН:**
- ❌ Модуль `core/voice_synthesizer.py` удалён (755 строк)
- ❌ Зависимости torch, torchaudio, pydub больше не требуются
- ❌ Команда `/voice` не работает
- ❌ Голосовые сообщения не отправляются
- ✅ Все персонажи общаются ТОЛЬКО текстом

**Причины удаления голосового режима:**
- Технические ограничения Silero TTS не позволили достичь естественного звучания
- Невозможность корректной обработки пауз и интонации
- Риск введения пользователей в заблуждение (ИИ звучит как человек)
- Высокое потребление ресурсов (~500MB зависимостей)
- Этические соображения (антропоморфизация ИИ)

---

### ⚠️ Общие важные предупреждения:

- В архиве есть живые секреты: токены Telegram, YooKassa и параметры БД. Их нужно считать скомпрометированными и ротировать.
- В логах тоже есть чувствительные данные. Это плохая практика и риск утечки.
- `requirments.txt` пустой, поэтому проект сейчас нельзя нормально поставить одной командой.
- Есть два разных контура хранения данных: SQLite внутри бота и MySQL/вебхук для уведомлений. Это надо явно документировать и поддерживать синхронно.
- Часть checkout-состояния живёт только в памяти. После рестарта сессия пользователя теряется.
- Файл `solidsimple.ru/public_html/info.php` опасен для публичного сервера.
- В проекте есть тестовые, debug и backup-файлы. Их нельзя выкладывать как production-код.

### Критично (почини срочно — деньги / репутация / безопасность)

1. `ВАШ_ХОСТИНГ/public_html/.env`, `logs/bot.log`, `bot.log`
   - Что не так: в архиве лежат реальные секреты, а в логах видны чувствительные данные, включая токены.
   - Почему это плохо: токен бота или платёжные секреты можно украсть и использовать для спама, перехвата уведомлений или подмены платёжного потока.
   - Как лучше сделать: немедленно ротировать все токены и пароли, удалить секреты из истории, включить redaction в логировании.

```python
SENSITIVE_KEYS = {"BOT_TOKEN", "YOOKASSA_SECRET_KEY", "DB_PASSWORD"}
```

2. `solidsimple.ru/public_html/info.php:1`
   - Что не так: публичный `phpinfo()`.
   - Почему это плохо: он раскрывает конфигурацию сервера, расширения, пути и часто помогает атакующему.
   - Как лучше сделать: удалить файл с сервера полностью.

3. `replay_yookassa.py:4-5`
   - Что не так: в коде жёстко прописаны webhook URL и secret.
   - Почему это плохо: секреты нельзя хранить в исходниках, особенно в utility-скриптах.
   - Как лучше сделать: брать значения только из `.env` или аргументов запуска.

```python
import os
WEBHOOK_URL = os.environ["WEBHOOK_URL"]
X_FORWARD_SECRET = os.environ["X_FORWARD_SECRET"]
```

4. `core/yookassa_webhook.py:11-18`
   - Что не так: файл импортирует `YOOKASSA_WEBHOOK_PORT`, `YOOKASSA_WEBHOOK_HOST`, `YOOKASSA_ALLOWED_IP_RANGES`, `YOOKASSA_FORWARD_SECRET` из `core.config`, но в `core/config.py` этих переменных нет.
   - Почему это плохо: webhook-контур может не стартовать вообще или упадёт при импорте.
   - Как лучше сделать: добавить переменные в конфиг и валидировать их при старте.

```python
YOOKASSA_WEBHOOK_HOST = os.getenv("YOOKASSA_WEBHOOK_HOST", "127.0.0.1")
YOOKASSA_WEBHOOK_PORT = int(os.getenv("YOOKASSA_WEBHOOK_PORT", "8080"))
YOOKASSA_ALLOWED_IP_RANGES = ["..."]
YOOKASSA_FORWARD_SECRET = os.getenv("YOOKASSA_FORWARD_SECRET", "")
```

5. `core/config.py:248-250`
   - Что не так: `PAYMENT_PROVIDER_TOKEN` захардкожен как `TEST:123456789`.
   - Почему это плохо: реальные Telegram Payments так не заработают, а ошибка будет скрыта за “тестовым” значением.
   - Как лучше сделать: убрать заглушку и брать значение только из `.env`.

```python
PAYMENT_PROVIDER_TOKEN = os.getenv("PROVIDER_TOKEN")
```

6. `core/database.py:21-27`, `handlers/*` и async-код вокруг них
   - Что не так: одна SQLite-связь используется с `check_same_thread=False` в многозадачном async-проекте.
   - Почему это плохо: при росте нагрузки вылезут гонки, блокировки и странные ошибки записи.
   - Как лучше сделать: перейти на нормальный pool/ORM, либо хотя бы открыть отдельное соединение на запрос и не шарить один connection на всё приложение.

   - Почему это плохо: compliance и privacy-риски.
   - Как лучше сделать: удалять/маскировать все связанные записи и делать это транзакционно.

### Мелочь / nice-to-have (улучшит жизнь)

1. `main.py.bak`, `__pycache__/`, `*.mp4`, логи и временные файлы в архиве
   - Что не так: в поставке слишком много мусора.
   - Почему это плохо: увеличивает путаницу и риск случайной публикации лишнего.
   - Как лучше сделать: добавить `.gitignore` и чистить артефакты перед деплоем.

2. `debug_logging.py`, `logging_patch.py`
   - Что не так: выглядит как дублирующая служебная логика.
   - Почему это плохо: поддерживать несколько мест с логированием не нужно.
   - Как лучше сделать: оставить один модуль logging-настройки.

3. `core/notifications.py`
   - Что не так: модуль слишком большой и делает слишком много.
   - Почему это плохо: его тяжело читать, тестировать и менять без побочных эффектов.
   - Как лучше сделать: разделить на `email`, `admin_alerts`, `shipping_notifications`, `formatters`.

4. Имена и стиль: `requirments.txt`, `DEESEEK_*`, длинные функции, частые `except Exception: pass`
   - Что не так: есть орфографические ошибки и слишком мягкая обработка ошибок.
   - Почему это плохо: скрывает реальные баги и снижает доверие к коду.
   - Как лучше сделать: привести имена к одному стилю, убрать пустые `except`, добавить нормальные исключения и докстринги.

   ```python
   # Вместо callback_data:
   InlineKeyboardButton("🏠 Главная", url="https://t.me/bot?start=home", style="primary")
   ```
   - Работает на ВСЕХ устройствах
   - Требует обработки `/start` параметров
   - Пользователь видит переход по ссылке
